# 검토 루프 지침

## 대원칙

1. 작성자는 자기 글의 결함을 보지 못한다. 검증은 작성자가 아닌 별도 관점이 수행한다.
2. 검토자는 서로 직교하는 실패 축을 맡는다. 축이 겹치면 검출력은 늘지 않고 중복 지적과 모순만 늘어난다.
3. 루프에는 종료 조건을 둔다. 검증 과정도 무한히 반복될 수 있다.

## 검토 패널

기본 패널은 세 검토자로 구성한다. 각 검토자의 설정, 평가 기준, 관할은 검토자 정의 파일을 모아 둔 reviewers 디렉터리의 개별 파일에 담는다.

- 검토자 A. 맥락 전문가 (reviewers/a-context-expert.md) — 장황함을 검출한다.
- 검토자 B. 무맥락 독자 (reviewers/b-blank-reader.md) — 설명 부족을 검출한다.
- 검토자 C. 표기, 구조, 문장 감사관 (reviewers/c-style-auditor.md) — 규칙 위반과 표기 오류를 검출한다.

각 파일의 "검토하지 않는 것" 항목이 검토자 사이의 경계를 정한다.

## 0단계. 기계 검사 게이트

판단이 필요 없는 검사는 검토자에게 맡기지 않는다.
LLM은 글자수를 정확히 세지 못하므로, 결정적 검사는 스크립트로 확정 판정한다.
검토자를 호출하기 전에 이 게이트를 먼저 통과시킨다.

1. **글자수 제한** — 제한이 있으면 반드시 실측한다. 추정하지 않는다.
   `python3 -c "print(len(open('파일').read().strip()))"`
2. **가운데점(·) 개수** — 0이어야 한다.
   `grep -o '·' 파일 | wc -l`
3. **금지어** — 규칙셋이 금지한 표현을 grep으로 확인한다.

하나라도 실패하면 수정한 뒤 게이트를 다시 통과시킨다.
전부 통과해야 1단계로 넘어간다.
게이트에서 걸린 글에는 검토자를 호출하지 않는다.

## 1단계. 실행 절차

1. 0단계 게이트를 통과한 초안에 대해서만 검증을 시작한다.
2. 세 검토자를 병렬로 실행한다. 각 검토자에게 해당 정의 파일을 지침으로 제공한다.
3. 검토자의 출력 형식을 강제한다.
   - 판정: PASS 또는 FAIL
   - FAIL인 경우 수정 지시 목록을 작성한다. 각 항목은 위치, 문제, 수정 제안으로 구성한다.
   - 지시 목록은 최대 10개로 제한하며, 사소한 취향 차이는 빼고 중요한 항목만 포함한다.
4. 전원 PASS인 경우 통과 처리하고 종료한다.
5. 한 명이라도 FAIL인 경우 지적사항을 반영하여 수정한 뒤 2번으로 돌아간다.
6. 최대 5회까지 반복한다. 5회에도 통과하지 못하면 사람에게 보고하고 판단을 받는다.
   검토자의 요구가 상충하는 경우에도 사람에게 보고한다.
   (예: A는 분량 축소를 요구하고 B는 설명 추가를 요구하는 경우)

## 운영 규칙

**역할 분리**

검토자는 글을 직접 수정하지 않는다. 검토자의 출력은 보고서 형태로만 수용한다.

**검사 제외 범위 고지**

인용문, 코드 블록, 프로그램 출력 등 원문 보존이 필요한 부분은
검사 대상에서 제외됨을 검토자에게 사전에 알린다.
고지하지 않으면 검토자가 원문 수정을 요구한다.

**동일 지적 통합**

같은 회차에서 여러 검토자가 같은 지점을 지적하면 하나로 합쳐 작성자에게 전달한다.

**반복 지적 방지**

재검증 시 검토자에게 이전 지적사항과 반영 내역을 전달한다.

**재검증 범위 조정**

문단 하나를 추가하는 수준의 국소 수정은 영향 범위에 해당하는 검토자만 실행한다.
전면 재작성은 세 검토자를 모두 다시 통과해야 한다.

## 금지 사항

- PASS 판정을 얻기 위해 검토 기준을 완화하지 않는다.
  기준 변경이 필요하면 사람의 승인을 받는다.
- 지적사항을 일부만 반영하고 재검증을 생략하지 않는다.
